Exceptions

Was ist eine Exception?

Eine Exception in Java ist ein Signal während der Programmausführung, das anzeigt, dass eine Ausnahmesituation eingetreten ist.

Beispiele für Ausnahmesituationen:

Es wird eine Referenz mit dem Wert null benutzt.
Es wird bei einem Array ein Index benutzt, der jenseits der Arraygrenzen liegt.
Ein Objekt ist nicht im Scope.
Eine Datei wird nicht gefunden.
Ein Netzwerk-Link ist fehlgeschlagen.

Das "throw-and-catch" Paradigma

Java liefert einen Mechanismus für einen systematischen Umgang mit Ausnahmesituationen. Der Mechanismus folgt dem throw-and-catch Paradigma:

"throw an exception"

Es wird eine Ausnahmesituation signalisiert. Das geschieht an der Stelle, an der die Ausnahmesituation festgestellt wird

"catch an exception"

Die Ausnahmesituation wird in geeigneter Form behandelt. Das geschieht an einer bestimmten anderen Stelle.

Warum Exceptions?

Kapselung der Fehlerbehandlung.
Trennung von Fehlererkennung vom Behandlungsmechanismus.
Exceptions dürfen nicht ingnoriert werden. Der Programmierer ist gezwungen, auf Exceptions zu reagieren.
Trennung von Programmlogik und Fehlerlogik.
Schnellere Abarbeitung von Programmteilen, wenn keine Fehler auftauchen, als mit herkömmlichen Fehlerabfragen, welche immer ausgeführt werden.

Exceptions und Exception-Handler

Eine "Exception werfen" heißt, eine dafür vorgesehene Anweisung ausführen, der man ein Exception - Objekt mitgibt, das die gesamte Information über die Ausnahmesituation enthält.

Eine "Exception auffangen" heißt, einen geeigneten Exception - Handler zu finden, dem die Ausführungskontrolle gegeben wird.

Dem Exception - Handler wird das "geworfene" Exception - Objekt übergeben.

Der Exception - Handler ergreift die geeigneten Maßnahmen zur Behandlung der Ausnahmesituation.

Die Exception Klassen - Hierachie

Die Exception - Klassen sind im Paket java.lang enthalten. Alle Exceptions sind von der Superklasse Throwable abgeleitet. Eine Konsequenz der Hierarchie ist, dass es ausreicht, einen Fehler in der Oberklasse aufzufangen. Wenn zum Beispiel eine FileNotFoundException auftritt, dann ist diese Klasse von IOException abgeleitet, was heißt, FileNotFoundException ist eine Spezialisierung. Wenn wir alle IOException auffangen, behandeln wir damit auch gleichzeitig die FileNotFoundException mit.

In der Regel sind Eceptions in Java so genannte checked Exceptions: Der Compiler stellt sicher, dass eine Methode, die eine checked Exception wirft (direkt oder indirekt), sich auch mit dieser Exception befasst.

Es gibt zwei Möglichkeiten, sich mit einer checked Exception zu befassen:

Man kann die Exception in einem Exception-Handler auffangen und bearbeiten, oder man kann die Exception an die aufrufende Methode zu Behandlung weitergeben.

Bei unchecked Exceptions unterlässt der Compiler die Prüfung. Dies ist  bei allen Exceptions der Fall, die von der Klasse Error und RuntimeException abgeleitet sind.

Von Error abgeleitete Exception-Klassen werden von der Laufzeitumgebung geworfen und sind in der Regel nicht behebbar.

Einige Fehlerarten können potenziell an vielen Programmstellen auftreten, etwa eine Division durch Null oder ungültige Indexwerte beim Zugriff auf Array-Elemente. Treten solche Fehler beim Programmlauf auf, liegt in der Regel ein Denkfehler des Programmierers vor, und das Programm sollte normalerweise nicht versuchen, die ausgelöste Ausnahme aufzufangen und zu behandeln. Dies sind Exceptions die von der Klasse RuntimeException abgeleitet sind.

Uncheked Exceptions können aufgefangen werden, es besteht aber kein Zwang dazu.

Um Ausnahmesituationen in eigenen Programmen spezifisch zu behandlen, implementiert man eigene Exception-Klassen. Eigene Exception-Klassen sollen von der Klasse Exception abgeleitet werden, damit sie checked Exception sind. Die Klasse Exception hat ein Attribut vom Typ String, das im Konstruktor mit dem Argument des Typs String gesetzt werden kann. Wenn man dies nicht ausnutzen will, dann benutzt man den parameterlosen Konstruktor von Exception.

Beispiel für eine eigene Exception-Klasse

 

public class MerkmaleException extends Exception

{

             public static final int bereitsVorhanden = 1;

             public static final int nichtVorhanden =2;

          

             public final String name;

             public final int grund;

             public MerkmaleException(String name, int grund)

             {

                  this.name = name;

                  this.grund = grund;

             }

 }

Exception Handling als Sprachbestandteil

Java unterstützt die Ausnahmebehandlung durch die folgenden Sprachelemente:

Die throw Anweisung : Mit dieser Anweisung wird eine Exception geworfen. Bei einer throw Klausel im Methodenkopf, muss eine Methode alle checked Exceptions ausführen, die die Methode wirft, aber nicht selbst auffängt.

Die try – catch Anweisung: Sie dient der Behandlung von Exceptions.

Die throw-Anweisung

Exceptions werden mit der throw-Anweisung geworfen.

throw <Objekt-Referenz-Ausdruck>

Das Resultat von <Objekt-Referenz-Ausdruck> muss eine Referenz sein, deren Typ zu Throwable kompatibel ist. throw unterbricht unmittelbar die Ausführung des Programms an der betroffenen Stelle und macht sich auf die Suche nach einem geeigneten Exception-Handler.

Die throw-Anweisung im Beispiel:

public void ersetzeWert(String name, Object wert) throws MerkmaleException

{

             if(!vorhanden(name))

             {

 

               throw new MerkmaleException(name,MerkmaleException.nichtVorhanden);

             }

             topf.put(name,wert);

 }

Die Methode ersetzeWert( ) im obigen Beispiel erklärt im Methodenkopf mit der throws-Klausel, dass sie eine checked Exception wirft und nicht selbst auffängt.

Die allgemeine Form der throws-Klausel ist:

throws Exceptiontyp_1, Exceptiontyp_2, ..........., Exceptiontyp_n

Der Compiler überprüft (checkt),ob alle in der Methode geworfenen und nicht behandelten checked Exceptions in der Klausel deklariert werden. Die aufrufende Mehtode muss sich um diese Exceptions kümmern.

Die try – catch Anweisung

try

{

             Anweisungen

}

catch

{

             Anweisungen

}

catch

{

             Anweisungen

}

...........

finally

{

             Anweisungen

}

Die try-Klausel enthält die Anweisungen die man möglichst ohne eine Exception durchbringen will.

In den catch-Blöcken befinden sich die Exception-Handler. Jeder catch-Block steht für einen eigenen Exception Typ. Jeder Exception-Typ darf nur einmal vorkommen. Es darf kein allgemeinerer Typ vor einem spezielleren kommen. Die catch-Blöcke sind optional. Wird kein catch-Block angegeben, muss die finally-Klausel vorhanden sein.

In der finally-Klausel befinden sich Anweisungen, die in jedem Fall ausgeführt werden. Üblicherweise wird hier ein eventuell notwendiges "Clean Up" durchgeführt. Die finally-Klausel ist optional. Ist keine finally-Klausel vorhanden, dann muss wenigstens eine catch-Klausel spezifiziert werden.

Beispiel:

String name = "Alter";

Object wert = new Integer(8);

try

{

             merkmale.ersetzeWert(name, wert);

}

catch (MerkmaleException me)

{

             if (me.grund == MerkmaleException.nichtVorhanden)

             {

              .............

             }

             else

             {

              System.out.println(me);

             }

}

Der Ablauf bei throw

throw unterbricht unmittelbar die Ausführung des Programms an der betroffenen Stelle und macht sich auf die Suche nach einem geeigneten Exception - Handler.Wird für die Exception in der Methode, in der sie geworfen wird, kein Exception - Handler gefunden, dann wird der Aufruf - Stack zurückgegangen, ständig auf der Suche nach einem geeigneten Exception-Handler in einer umgebenden try-Anweisung. Wird auf diesem Weg kein geeigneter Exception - Handler gefunden, landet die Exception schließlich beim so genannten Default Exception-Handler. Grundsätzlich soll man Exception selber behandeln bzw. höchstens zwei Ebenen hinaufgeben, dann muss eine Behandlung sichergestellt sein.

Der Default Exception-Handler gibt eine „erklärende Nachricht“ aus, gefolgt von einem Stack - Trace. Eine nicht aufgefangene Exception (uncaught exception) resultiert in der Beendigung des Threads, in dem die Exception geworfen wurde.

Die Ausgabe des Stack – Trace kann man auch selber über die Mehode printStackTrace( ) des Exceptions - Objektes erzwingen.

Die throws-Klausel beim Überschreiben von Methoden

Beim Überschreiben einer geerbten Methode oder beim Implementieren einer geerbten abstrakten Methode muss die throws - Klausel der überschreibenden bzw. implementierenden Methode zu der throws - Klausel der geerbten Methode kompatibel sein, d.h. sie darf keine zusätzlichen checked Exceptions spezifizieren, sie darf aber welche weglassen.
